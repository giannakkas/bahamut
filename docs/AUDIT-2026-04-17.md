# Bahamut.AI — Adversarial Production-Readiness Audit

**Repo:** `giannakkas/bahamut`
**Audit date:** 2026-04-17 (commit at the time of clone)
**Auditor stance:** Hedge-fund production-readiness, exchange execution systems, SRE pre-launch kill review. Every assumption treated as unsafe until verified in code. Real capital at risk.

---

## 0. TL;DR

> **Bahamut is a strategy platform with a live-broker wrapper bolted on. It is not a production-grade execution engine. Do not wire real money to it in its current state.**

The strategy, data, and indicator layers are structurally sound and in some areas above typical hobby standard (closed-candle enforcement, Wilder ADX, synthetic-data blocking, interval-correct realized-vol annualization, tiered strategy ensemble). The production-grade infrastructure around them — idempotency, broker truth, reconciliation, multi-worker safety, kill-switch semantics, operator trust, schema discipline — is largely aspirational. There are four parallel execution engines, a state machine that was built but never wired, a reconciler that detects orphans and then logs them, a broker adapter that never sends a `newClientOrderId`, a duplicate-trade generator hiding in `main.py`, and a WS authenticator that accepts any string that looks like a JWT. Real capital would be lost to operational defects, not strategy defects.

This document walks the full lifecycle in code, enumerates 71+ findings with file/function precision, ranks the top 20 blocking items, gives exact patches, and closes with the end-state architecture I would build if I wanted Bahamut to be best-in-class.

## 2. Architecture — the REAL trading flow

What the README says is not what the code does. Here is the actual lifecycle, traced in code.

### 2.1 Signal generation (every 10 minutes, three concurrent sources)

```
┌─ Celery Beat ─────────────────────── (if configured in Railway env)
│  schedule: training-cycle 600s
│  → run_training_cycle()
│
├─ Uvicorn worker 1 ─ main.py:_training_cycle_loop (asyncio.to_thread)
│  schedule: every 600s — INSIDE the API process
│  → run_training_cycle()  (SAME function)
│
└─ Uvicorn worker 2 ─ main.py:_training_cycle_loop (asyncio.to_thread)
   schedule: every 600s — INSIDE the API process
   → run_training_cycle()  (SAME function)
```

Per Chris's documented deployment constraint (`--workers 2`), the training cycle runs three times per nominal 10-minute interval. Dedup is partly preserved by the Redis idempotency key on `signal_id`, but `signal_id = f"{asset}:{strategy}:{direction}:{trigger_reason}:{int(time.time() // 600)}"` is bucketed on wall clock and is not stable across cycle-boundary drift.

|---|---|---|---|---|

## 4. Accuracy Risks

### 4.1 Candle-derived PnL survives broker-close failure — **CRITICAL**

`training/engine.py:update_positions_for_asset`. Flow (lines ~1358–1440):

1. Candle SL/TP/TIMEOUT triggers → `exit_price` set from `pos.stop_price | pos.tp_price | bar.close`.
2. `trade = TrainingTrade(..., pnl=round(pnl, 2), ...)` populated from candle `exit_price`.
3. `_remove_position(pos.position_id)` — local position removed.
4. `exec_close(...)` — broker close attempted.
5. If `exec_close` succeeds and returns `fill_price > 0`: `trade.exit_price` and `trade.pnl` are overwritten with broker-truth.
6. If `exec_close` **raises or returns fill_price = 0** (timeout, 5xx, internal fallback, `_wait_for_fill` fails on Alpaca sells): the candle-derived PnL from step 2 is what `_persist_trade(trade)` writes to `training_trades`. The broker position remains open.

The existing audit (`docs/AUDIT-2026-04-17.md`) claims broker-truth PnL; that's only true on the happy path. **Every exec_close failure persists a fictional candle PnL and leaves broker exposure open.**

### 4.2 Realized risk ≠ intended risk — **HIGH**

`training/engine.py:open_training_position` (~line 1010–1090). After broker fill returns `exec_result["fill_price"]`, the code:

```python
pos.entry_price = round(exec_result["fill_price"], 6)
pos.stop_price = round(pos.entry_price * (1 - sl_pct), 6)
pos.tp_price   = round(pos.entry_price * (1 + tp_pct), 6)
# pos.size is NOT recomputed
```

`pos.size` was computed from the *pre-fill* entry price (`size = risk_amount / max(0.01, entry_price * sl_pct)`). If the broker filled 20 bps worse, actual `(fill - sl_pct_of_fill) * size` ≠ `risk_amount`. Over 1000 trades you systematically over- or under-risk by the mean slippage.

### 4.3 Cross-broker balance summed — **HIGH**

`execution/balance.py:get_real_balance`. Binance Futures equity (USDT) and Alpaca equity (USD) are added into `combined_usd`. `get_available_risk(0.02)` returns `combined * 0.02`. Training orchestrator uses this number for every asset class.

Scenario: $25K Alpaca, $10K Binance → combined $35K → `max_risk_usd = $700`. A crypto trade sized for $700 of risk is 7% of Binance equity, not 2%. Over a string of losing crypto trades you can vaporize the Binance account while Alpaca looks fine.

### 4.4 Unrealized PnL folded into sizing base — **HIGH**

`execution/balance.py` line ~65:
```python
combined += total + unrealized
```
Binance Futures `totalUnrealizedProfit` is included in the equity number used for sizing. As winners run, new trades get sized larger — silent, asymmetric pyramiding with no strategy framework behind it.

### 4.5 API hiccup → virtual-capital fallback → 10× over-sizing — **CRITICAL**

`execution/balance.py` line ~95:
```python
if combined <= 0:
    combined = TRAINING_VIRTUAL_CAPITAL  # 100_000
    result["source"] = "virtual_default"
```

If Binance returns a partial response (connection drop mid-body) *and* Alpaca also blanks, combined = 0, fallback to $100K. One cycle of every-trade-sized-against-$100K-instead-of-actual-$10K. `source = "virtual"` is the only signal; the orchestrator does not refuse to trade on it.

### 4.6 Candle SL/TP snap — gap unhandled — **HIGH**

`execution/paper_broker.py` and `training/engine.py:update_positions_for_asset`. If a bar's `low <= pos.stop_price`, `exit_price = pos.stop_price`. If the bar actually *opened* 3% below SL (weekend gap), real fill is the open. Both backtest and paper snap to the SL level, systematically generous to the trader. Live path takes real broker fill when broker is reachable — so backtest and live diverge exactly when gaps happen.

### 4.7 Commission asymmetry — **MEDIUM**

- Paper broker and backtest: 0.0004 (4 bps) round-trip per side.
- Binance Futures `/order` response often returns 0 commission; real commission requires `/userTrades`. Current code reads `exec_result.get("commission", 0)`. When zero, PnL is un-discounted.

### 4.8 Sub-dollar precision lost — **LOW but universal**

`execution/paper_broker.py`: `round(fill_price, 2)` on SHIB, PEPE, BONK collapses prices to $0.00. These assets can't paper-trade correctly. They're also blocked by `PRICE_SANITY["crypto"] = (0.0001, 200000)` in training — but if any bypass exists, the rounding bug is silent.

### 4.9 `pnl_pct` vs `return_pct` misnaming — **MEDIUM (operator trust)**

Training engine computes:
- `pnl_pct = pnl / risk_amount`  → this is R-multiple
- `return_pct = pnl / (entry_price * size)` → this is notional return

Both are labeled "pct" in the code and in the UI. A `pnl_pct = 2.0` is "+2R" (good), not "+200%". Easy operator misread.

### 4.10 Exit slippage = 0 in sim — **MEDIUM**

Paper broker and backtest apply slippage/spread to entries only. Exits snap to level. Live takes real broker fill. Sim overstates net PnL by ~1 spread-widths per trade.

## 5. Strategy Weaknesses

### 5.1 `_ema200_degraded` flag never consulted — **HIGH**

`features/indicators.py` sets the flag when fewer than 200 candles are available, logs a warning, and returns a shorter-period EMA. `grep` confirms **zero** strategies gate on this flag. v5_base `close <= ema_200` gate, v10 `dist_ema200_pct` calc, and v8 regime detector `dist_to_ema200_pct` thresholds all consume a potentially degraded value without knowing.

### 5.2 `_adx` default = 20 on insufficient data

Safe for `adx > 25` filters (blocks). Unsafe for `adx < 20` ranging-regime filters (fires on no data). Need a `_adx_degraded` flag with the same pattern as EMA-200.

### 5.3 v5_base — lagging, single-factor, LONG-only

- EMA20 × EMA50 cross is a lagging signal; by the time it fires, the move is 1–2 bars in.
- No volume confirmation, no momentum confirmation, no structure confirmation.
- Fixed quality score 0.7 regardless of how strong the cross is.
- No SHORT capability (death cross would be the mirror edge).

### 5.4 v9_breakout — LONG-only, volume-blind, short lookback

- 20-bar breakout on 15-m = 5-hour high. Noise-level on most crypto.
- 0.995 tolerance lets "confirmation" close 0.5% below the level.
- No volume confirmation — range expansion alone is not demand.
- Breakdowns are not captured.

### 5.5 v10_mean_reversion — crypto RANGE disabled, crash-SHORT narrow

- Crypto RANGE LONGs explicitly blocked for −0.12 expectancy.
- Crypto CRASH SHORT entry window is very narrow (rally within 0.5% of EMA20 + RSI 35–70 + rejection + 2/3 declining). Fires rarely; statistically weak.
- Mean reversion on fat-tailed 15m crypto distributions is structurally fragile.

### 5.6 Cross-strategy weaknesses

- At most one trade per asset per cycle (dedup at asset level). Simultaneous v5 + v9 signals on BTC produce one trade.
- No correlation-aware portfolio sizing. Two highly correlated crypto trades count as independent.
- No event avoidance. Strategies are purely technical — no FOMC, CPI, halving, or unlock blackout.
- Multiplier stacking is unbounded: `risk_amount *= re_mult * news_mult * ai_candidate_mult * ai_global_mult * crash_short_penalty`. Five independent multipliers, each clamped to [0, 1]. With independent distributions you can legitimately get risk_amount → 0 on a healthy signal or collapse to 0.1 × intended without clear operator visibility.

## 7. Execution & Broker Integration Weaknesses

### 7.1 `newClientOrderId` is never sent — **CRITICAL**

`execution/binance_futures.py:place_market_order` and `execution/alpaca_adapter.py:place_market_*`. Both adapters build their own `client_order_id` in `OrderManager.create_intent` (`bah_{asset}_{ts}_{uuid8}`) and then never send it to the exchange. Without it:
- On timeout you cannot query `/fapi/v1/order?origClientOrderId=…` to see if the order was accepted.
- You cannot safely retry — Binance will accept a duplicate.
- Alpaca cannot de-dup either.

### 7.2 Timeout ≡ rejection — **CRITICAL**

Both adapters end with `except Exception as e: return {"error": str(e)}`. Caller cannot distinguish "server rejected" from "server never responded." Training engine's `open_training_position` treats both as a clean abort when `_expected_platform != "internal"`, but the order may be working on Binance after a timeout.

### 7.3 No `reduceOnly` on closes — **CRITICAL**

`_binance_close` does `side = "SELL" if direction == "LONG" else "BUY"`. A race with reconciliation or a half-closed partial fill could flip the position instead of flattening it. `reduceOnly=true` is a one-line fix that makes the close idempotent at the broker.

### 7.4 State machine unused — **CRITICAL**

`order_manager.py` defines `INTENT_CREATED → SUBMIT_PENDING → SUBMITTED → ACKNOWLEDGED → PARTIALLY_FILLED → FILLED → CLOSED` plus `SUBMISSION_UNKNOWN` and `RECONCILE_REQUIRED`. Production callers only ever call `create_intent`. `record_fill`, `record_close`, and non-trivial `transition()` calls exist only in `tests/test_e2e_production.py` and one shutdown-path transition to `RECONCILE_REQUIRED`. Consequences:
- `order_intents.order_state` stays at `intent_created` forever.
- `order_intents.position_state` stays at `opening` forever.
- `order_events` audit log is empty (only initial row inserted).
- `reconciliation.get_open_intents()` filters `position_state='open'` and returns nothing → falls back to `_load_positions()` (the very thing the state machine was meant to replace).

### 7.5 Reconciliation: detect-only — **CRITICAL**

`execution/reconciliation.py`:
- Orphans (broker has position, local has nothing): appended to `result["orphans"]` and logged. No force-close, no alert escalation, no bracket-order audit, no Telegram.
- Missing (local says open, broker has nothing): same — log only. Could be a broker-side SL/TP hit (which means Bahamut should record the PnL from broker user-trades) or a never-filled order. Current code doesn't disambiguate and doesn't close the local record; `update_positions_for_asset` will continue marking it to market and eventually synthesize a candle PnL.
- Mismatches (qty/side differ): marked `RECONCILE_REQUIRED` in the cosmetic state machine, which goes nowhere.
- Cadence: every 6th cycle (~1 hr). In crypto futures you can be liquidated in 60 minutes.
- Symbol reverse-map `symbol.replace("USDT", "USD")` breaks on anything with USDT in the base.
- Assumes one-way mode. Hedge-mode accounts report LONG and SHORT as separate rows with `positionSide`. Current code silently collapses them.
- `local_positions` is keyed by asset. Multi-strategy same-asset positions collapse.

### 7.6 Circuit breaker blocks closes — **CRITICAL**

`execution/router.py:execute_close` returns `circuit_breaker_open_on_close` when OPEN. A tripped breaker LOCKS positions in — the exact opposite of what you want. The comment acknowledges the problem then the code ships with the bug anyway. Closes must always be allowed through.

### 7.7 Circuit breaker is global, single-platform — **CRITICAL**

`execution/circuit_breaker.py`: one `CircuitBreaker()` singleton counts failures from Binance + Alpaca together. Five Binance hiccups disable Alpaca too. And Redis state is restored only in `__init__` — cross-worker visibility is promised in the docstring but not delivered at runtime.

### 7.8 Live/demo is a single env var — **HIGH**

`BINANCE_FUTURES_BASE_URL` defaulted to demo; one env-var flip goes to production. There is no `ENABLE_LIVE_TRADING=1` cross-check, no startup-time assertion pairing the base URL with the key type. One typo or a copy-paste from staging to production gets you live.

### 7.9 Alpaca asymmetry — **CRITICAL**

`place_market_buy` polls `_wait_for_fill` (up to 5 × 1 s synchronous `time.sleep`). `place_market_sell` does not. Closes return `fill_price=0` until the next reconciliation. `update_positions_for_asset` then writes candle PnL and removes the local position. After-hours stock opens queue at Alpaca as `day` TIF and become orphans locally.

### 7.10 No `recvWindow`, no leverage/margin-mode startup check, no hedge-mode handling

`execution/binance_futures.py` signs with system clock (`time.time() * 1000`), no `recvWindow` parameter. Railway clock drift sporadically throws -1021. No call to `/fapi/v1/leverage` to normalize per-symbol leverage at startup; no margin-mode verification (cross vs isolated). `positionSide` omitted on orders → hedge-mode unsafe.

### 7.11 Silent live → paper degradation

`execution/router.py:_get_platform` returns `"internal"` whenever the adapter's `_configured()` is False. Missing `BINANCE_FUTURES_API_KEY` in Railway env produces internal-sim trades without any warning; `_expected_platform` is derived from the same function and so doesn't trip the guard in `open_training_position`.

| TP exit | `tp_level` (no slip) | `tp_level` (no slip) | broker fill (real slip) |
| Gap through SL | snap to SL | snap to SL | real gap fill |
| Commission | 0.0004 round-trip | 0.0004 round-trip | broker-reported, often 0 on `/order` |
| Sizing base | `self.balance` (compounds) | `sleeve_equity` param | `get_available_risk()` 60s cache, cross-broker sum |
| Concurrent workers | single-threaded | single-threaded | 2–3 workers race the same signal |

**Verdict: materially divergent.** The pyramid strategy only exists in backtest. Exit slippage is zero in sim and real in live. Meme-coin spreads are 50× sim assumption. Any PnL metric computed from backtest is not an estimator of live performance.

## 11. Railway / WebSocket / Distributed Systems Weaknesses

### 11.1 Training cycle runs 3× — **CRITICAL**

`main.py:lifespan` starts `_training_cycle_loop` per worker. `celery_app.py:beat_schedule` has `training-cycle: schedule: 600.0`. With Celery Beat running and `--workers 2`, the cycle executes from 3 concurrent sources every 600 s.

### 11.2 In-memory singletons everywhere — **CRITICAL**

| State | Location |
|---|---|
| `ExecutionEngine.open_positions` | `execution/engine.py` |
| `ExecutionEngine._kill_switch` | `execution/engine.py` |
| `ExecutionEngine._processed_signals` | `execution/engine.py` |
| `CircuitBreaker` counters after init | `execution/circuit_breaker.py` |
| `_shutting_down` | `execution/shutdown.py` |
| `_manual_override` | `portfolio/kill_switch.py` |
| `open_training_position._opened_this_cycle` | function attribute |
| `update_positions_for_asset._already_closed` | function attribute |
| `AdminConnectionManager.clients` | `ws/admin_live.py` |

Every one of these is per-process. Railway restarts, scale-ups, and the `--workers 2` setup all break the assumed shared state.

### 11.3 WS fanout is "single replica" explicitly — **CRITICAL**

`ws/admin_live.py` docstring: *"In-process event bus (single replica on Railway)"*. With 2 workers, every event fires on one worker's subscribers only. A dashboard client sitting on the other worker misses up to half of all events. No Redis pub/sub, no sequence numbers, no resume tokens.

### 11.4 WS auth bypass — **CRITICAL SECURITY**

`ws/admin_live.py:_authenticate` final fallback:
```python
# Last resort: if token looks like a JWT, accept it
# (admin panel already verified auth before showing the page)
parts = token.split(".")
if len(parts) == 3:
    payload = _json.loads(base64.urlsafe_b64decode(padded))
    if payload.get("sub") or payload.get("email"):
        return payload
```
Any string of the form `a.b.c` where `b` is a valid base64 JSON with a `sub` or `email` field is accepted. No signature verification. Also, `{"verify_exp": False}` is in the opts loop — expired tokens accepted.

### 11.5 `publish_event` spins event loops in worker threads

When called from a Celery task or threadpool thread without an event loop, `publish_event` does `loop = asyncio.new_event_loop(); loop.run_until_complete(...); loop.close()` per event. Under load this churns file descriptors and GC. Exceptions are swallowed.

### 11.6 Shutdown signal is local

`is_shutting_down()` reads the in-process `_shutting_down` only. The Redis `bahamut:shutdown_in_progress` marker is written but never read. Worker A's SIGTERM doesn't reach Worker B until Worker B itself gets SIGTERM.

### 11.7 Reconciliation cadence is too slow

Hourly. Crypto futures liquidation can happen in minutes. A Railway restart while in flight → 1-hour exposure window with no local SL/TP.

### 12.1 Schema is not versioned — **CRITICAL**

One Alembic migration (`001_initial.py`) calls `Base.metadata.create_all()`. `OrderManager._log_event` swallows errors. `order_events` is almost always empty anyway because no transitions happen in prod. There is no dispute-resolution-grade record of what was submitted, what the broker responded, what was filled, or what was closed.

### #5 — Reconciliation: auto-correct, not log-only; per-minute cadence

`execution/reconciliation.py`:

Orphan (broker has, local doesn't):
1. Check `order_intents` for intent in `SUBMISSION_UNKNOWN` with matching asset/direction/qty → promote to `FILLED` via `record_fill` from broker data.
2. If no matching intent → treat as unauthorized position: Telegram/email alert + SAFE MODE flag + optional auto-close (configurable).

Missing (local has, broker doesn't):
1. Fetch broker `/userTrades` and `/allOrders` for the client_order_id.
2. If broker-side SL/TP closure found → `record_close` with that fill data.
3. Else → mark `SUBMISSION_UNKNOWN` and re-submit close with `reduceOnly=true`.

Cadence: every cycle (not every 6th). Cheap: two API calls per broker.

**Principles:**
1. **Broker is always the source of truth.** Local state is a cache of broker state, reconciled continuously.
2. **Every state change is an event.** Orders, fills, positions, commissions, closes — every one appends to an immutable log. PnL is a derived view over this log. Disputes are resolvable.
3. **Every order carries an idempotency key** (`newClientOrderId` + our own `signal_id`). Retries are safe. Timeouts are survivable. Network partitions do not create ghosts.
4. **SL/TP is broker-side** via bracket orders. Survives app restarts. Survives network outages. Survives cloud-provider incidents.
5. **State lives in Redis and Postgres, never in process memory.** Every worker is replaceable. Scale-out is linear.
6. **Kill switch is atomic and global.** One Redis flag, one endpoint, all engines observe, all workers obey, all broker positions flat.
7. **Operator console shows broker truth.** Dashboard state is a subscription to the same stream the engines subscribe to. When it says "LIVE", every one of a specific list of checks is green.
8. **Schema is Alembic-only.** CI fails PRs that add runtime DDL.
9. **Security is explicit.** Live vs demo is a signed config artifact, not an env var. Admin actions require 2FA. WS auth verifies signatures.
10. **Backtest, paper, and live share the same fill model.** Per-asset slippage tables calibrated from realized fills feed all three. Paper trading is a runtime mode of the live engine, not a separate module.
