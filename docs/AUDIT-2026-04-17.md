# Bahamut.AI — Production Trading Readiness Audit

**Date:** April 17, 2026  
**Scope:** Full repository analysis — giannakkas/bahamut  
**Verdict:** NOT production-ready. Paper trading is functional with meaningful improvements already made. Several critical gaps remain before real capital.

---

## 1. Executive Summary

Bahamut is a surprisingly well-structured multi-strategy algorithmic trading platform. It shows clear evidence of iterative hardening — closed-candle enforcement, synthetic data blocking, engine-level suppress maps, dual-timeframe regime detection, and sentiment overlays. The codebase is far more mature than a typical hobby project.

**However, it is not ready for live capital.** The core issues are:

1. **No trading fees modeled anywhere** — PnL is overstated on every single trade
2. **ADX calculation is mathematically wrong** — it computes a single unsmoothed DX, not the smoothed average that ADX is
3. **Kill switch closes positions at a single `price` parameter** — not per-asset mark-to-market prices
4. **EMA-200 uses degraded fallback** when fewer than 200 candles are available — strategies silently operate on incorrect values
5. **Backtesting and paper trading share no common execution path** — results are structurally non-comparable
6. **Position state lives in a Python singleton** — a process restart between DB reconciliation and the next cycle creates a window for orphan/duplicate positions
7. **Realized volatility uses √252 annualization for 15-minute crypto candles** — a 30x overstatement

The good news: data provenance is solid (Binance public API, closed-candle enforcement, synthetic blocking), the strategy logic is structurally sound (three complementary strategies covering TREND/RANGE/BREAKOUT), and the risk/execution policy layer is thorough. The platform is 70-80% of the way to being trustworthy.

## 3. Accuracy Risks (Critical)

### 3.1 No Fee Model — Severity: CRITICAL

**Files:** `execution/paper_broker.py`, `training/engine.py`

The paper broker applies slippage (8 bps) and spread (12 bps) but **zero commission fees**. Binance Futures charges 0.02% maker / 0.04% taker. On a round-trip trade, that's 0.04-0.08% of notional value. For a system targeting 2-5% TP on 15m candles, this is a **1-4% drag on net profit per trade** that is completely invisible.

**Impact:** Every PnL number, win rate, expectancy, and Sharpe ratio is overstated. A strategy showing +2% average win after 2.5% TP is actually +1.92% after fees. A strategy showing -1.5% average loss after 2.5% SL is actually -1.58%. This systematically inflates profit factor.

**Fix:**
```python
# paper_broker.py — add to _close()
COMMISSION_RATE = 0.0004  # 0.04% taker (round-trip)
notional = exit_price * pos.size
commission = notional * COMMISSION_RATE * 2  # entry + exit
pnl = pnl - commission
```

### 3.2 ADX Calculation Is Wrong — Severity: HIGH

**File:** `features/indicators.py`, line 216

The `_adx()` function computes +DI, -DI, then a single DX value and returns it as "ADX". **This is not ADX.** ADX is the smoothed (Wilder) average of DX over `period` bars. The current implementation returns a single-point DX, which is much noisier and numerically different from true ADX.

The code comments say "ADX" and the key is `adx_14`, but mathematically it's just DX. This means:
- The ADX < 20 filter in v5_base is actually a DX < 20 filter (noisier, more false triggers)
- The regime detector's `adx > 25` threshold is calibrated against DX, not ADX

**Fix:** After computing DX for each bar, apply Wilder smoothing over `period` bars to get the true ADX. Alternatively, acknowledge it as DX and recalibrate thresholds.

### 3.3 EMA-200 Degraded Fallback — Severity: HIGH

**File:** `features/indicators.py`, line 88

```python
if len(closes) >= 200:
    result["ema_200"] = _ema(closes, 200)
else:
    result["ema_200"] = _ema(closes, len(closes) - 1) if len(closes) > 1 else closes[-1]
```

When the system has only 200 × 15m = 50 hours of candles (the `CRYPTO_CANDLE_COUNT = 200`), it often has exactly or fewer than 200 closed candles after dropping the forming bar. In that case, EMA-200 degrades to `_ema(closes, 199)` or less. This is a materially different indicator.

**Impact on strategies:**
- v5_base: `close <= ema_200` gate becomes unreliable → false entries or missed entries
- v10 crash_short: `dist_ema200_pct > 0.02` check uses a wrong reference → SHORTs on non-crash assets
- v8 regime detector: `dist_to_ema200_pct` thresholds (±3%, ±5%, ±8%) are calibrated for true EMA-200

**Fix:** Increase `CRYPTO_CANDLE_COUNT` to 300 (300 × 15m = 75 hours — cheap, Binance allows up to 1000). Alternatively, flag when EMA-200 is degraded and skip strategies that depend on it.

### 3.4 Realized Volatility Annualization Wrong for Crypto 15m — Severity: MEDIUM

**File:** `features/indicators.py`, line 102

```python
result["realized_vol_20"] = float(np.std(returns) * np.sqrt(252))
```

√252 is the annualization factor for **daily** bars (252 trading days). For 15-minute bars on 24/7 crypto, the correct factor is √(365 × 24 × 4) = √35,040 ≈ 187. Using √252 ≈ 15.87 understates volatility by ~12x if the candles are 15m. This indicator isn't directly used for trade decisions currently, but it feeds into portfolio intelligence and could silently miscalibrate risk models.

**Fix:** Pass the interval to `compute_indicators()` and compute the correct annualization factor.

### 3.5 Bollinger Bands Use Population Std — Severity: LOW

**File:** `features/indicators.py`, line 185

`np.std(closes[-period:])` computes population standard deviation (divides by N). Conventional Bollinger Bands use sample std (divides by N-1, i.e., `ddof=1`). For period=20, the difference is ~2.5%. Minor but creates slight divergence from standard charting platforms, which complicates validation.

## 4. Strategy Weaknesses

### 4.1 v5_base — EMA Cross Strategy

**Strengths:** Clean implementation, dynamic SL/TP by timeframe, ATR and ADX filters, EMA gap filter, suppress check.

**Weaknesses:**
- **Single signal only** — a golden cross with bull regime. No volume confirmation, no momentum confirmation, no structure validation. A false cross in a choppy market triggers a trade.
- **No SHORT capability** — the strategy is structurally limited to buying in uptrends. It does nothing in half of market conditions.
- **Fixed quality score (0.7)** — every signal gets the same quality regardless of how strong the cross is, how aligned the momentum is, or how clean the structure is.
- **EMA cross is a lagging signal** — by the time EMA20 crosses EMA50, the move has already started. Entries are systematically late.

**Recommendations:**
- Add volume expansion confirmation (volume > 1.5× 20-bar SMA on cross bar)
- Add MACD histogram confirmation (positive and rising on cross bar)
- Graduate quality score based on confirmation count (0.5 base + 0.1 per confirmation)
- Consider adding death cross (SHORT) capability

### 4.2 v9_breakout — Confirmed Breakout

**Strengths:** Multi-bar confirmation (3 bars), ATR-aware adaptive SL/TP, structural cap (invalidation level), range expansion check, not-too-extended filter.

**Weaknesses:**
- **LONG only** — breakdowns (confirmed breaks below support) are the mirror edge and are not captured.
- **No volume confirmation** — the strategy checks range expansion but not actual volume. In crypto, range expansion without volume is often a low-liquidity gap, not genuine demand.
- **20-bar lookback is short** — on 15m candles, 20 bars = 5 hours. A 5-hour high breakout is noise-level on most crypto assets. Consider 50-100 bars for more meaningful structural levels.
- **0.995 tolerance on confirmation** — allowing closes 0.5% below the breakout level is generous. A true confirmed breakout should hold at or above the level.

### 4.3 v10_mean_reversion — Range Bounce / Crash Short

**Strengths:** Most sophisticated strategy. Regime-aware, multi-condition entry (stretch + Bollinger + RSI + bounce), SHORT capability, crash-specific detection with multi-bar confirmation, quality scoring with multiple factors.

**Weaknesses:**
- **Crypto RANGE LONGs disabled** — the code explicitly blocks them due to negative expectancy (-0.1246). This means v10 for crypto only operates as CRASH SHORT. That's a very specific, infrequent pattern.
- **Mean reversion in crypto is structurally fragile** — crypto doesn't have a "mean" in the traditional sense. Bollinger band bounces in 15m crypto are unreliable because the distribution has fat tails.
- **Crash SHORT entry window is narrow** — requires rally to within 0.5% of EMA20 + RSI 35-70 + rejection candle + 2/3 bars declining. This may fire too infrequently to be statistically meaningful.

### 4.4 Cross-Strategy Concerns

- **All three strategies produce at most 1 trade per asset per cycle** — no pyramiding in the training system. If BTC flashes all three signals simultaneously, only one trade opens (dedup by asset).
- **No portfolio-level hedging** — if BTC is LONG via v5 and ETH shows a crash SHORT via v10, these positions are treated independently. No correlation-based hedge optimization.
- **No event avoidance** — all three strategies are purely technical. They cannot avoid trading into FOMC, CPI, or halving events.

## 5. Data Weaknesses

### 5.1 Single-Provider Risk for Stocks

Stocks still rely on Twelve Data as the sole data source. If Twelve Data has an outage or returns stale data, the system has no fallback except cached data (which degrades to `stale_cache` mode but is still served to strategies).

**Crypto is well-handled** — Binance public API is free, no rate limits, with closed-candle enforcement. This is genuinely solid.

### 5.2 No Bid/Ask Spread Data

The system uses mid-price (close) for everything. It doesn't fetch or model bid/ask spreads. The paper broker applies a fixed 12 bps spread, but real spreads vary dramatically:
- BTC/USDT: ~1-2 bps on Binance
- PEPE/USDT: ~5-20 bps depending on liquidity
- Low-cap tokens: 20-100+ bps

Using 12 bps fixed overcharges BTC and undercharges low-cap assets, distorting per-asset PnL comparison.

### 5.3 No Gap Detection

There's no validation that candles are contiguous. If Binance returns candles with a gap (missing bars due to exchange maintenance or API issues), the indicator calculations are silently wrong — EMA/ATR continuity is broken by gaps.

### 5.4 200-Candle Crypto Window Is Tight

`CRYPTO_CANDLE_COUNT = 200` × 15m = 50 hours. EMA-200 needs 200 bars to be properly seeded. After dropping the forming candle, you have 199 or fewer closed candles. This means EMA-200 is almost always running in degraded mode for crypto.

## 6. Backtesting Weaknesses

### 6.1 Completely Separate Code Path

The backtesting module (`backtesting/replay_v6.py`) has its own `V6Engine` with its own `Position` dataclass, its own fill logic, its own pyramid system. **None of this code is shared with the live training engine.** This means:

- Backtest results are structurally non-comparable to paper trading results
- Bug fixes in one path don't propagate to the other
- The backtester has features (pyramiding, trailing stops, profit locks) that the training engine doesn't use

### 6.2 Synthetic Data Still Exists in Backtester

`backtesting/data_loader.py` still has `generate_synthetic_candles()` with `np.random.seed(42)`. While the live system blocks synthetic data (`BAHAMUT_BLOCK_SYNTHETIC=1`), the backtester has no such guard. Someone running a backtest without an API key gets synthetic data without warning.

### 6.3 No Fee Model in Backtester Either

The V6Engine applies slippage and spread but zero commission. Same issue as the paper broker.

### 6.4 SL/TP Fill Assumptions

Both the paper broker and backtester fill SL at exactly the stop price and TP at exactly the take-profit price. In reality:
- SL fills gap through the stop — the actual fill is worse than the stop price
- TP fills can slip, especially in low liquidity
- Both broker and backtester assume "if SL and TP both hit on same bar, fill SL" — this is conservative for SL (correct) but the probability depends on which was hit first (unknown without tick data)

## 7. Execution and Risk Weaknesses

### 7.1 Kill Switch Uses Single Price — Severity: CRITICAL

**File:** `execution/engine.py`, line 164

```python
def activate_kill_switch(self, price: float):
    for pos in list(self.open_positions):
        trade = self.broker.force_close(pos, price, "KILL_SWITCH")
```

The kill switch takes a single `price` parameter and closes ALL positions at that price. But positions span different assets (BTC, ETH, SOL, AAPL, etc.). Closing a BTC position at the ETH price produces wrong PnL. Closing an AAPL position at the BTC price is nonsensical.

**Fix:** The kill switch must fetch current price per asset:
```python
def activate_kill_switch(self):
    for pos in list(self.open_positions):
        price = get_current_price(pos.asset)  # per-asset lookup
        trade = self.broker.force_close(pos, price, "KILL_SWITCH")
```

### 7.2 In-Memory Singleton State — Severity: HIGH

**File:** `execution/engine.py`

The `ExecutionEngine` is a Python singleton. All state (pending orders, open positions, closed trades) lives in memory. The `_reconcile_from_db()` function rebuilds state from PostgreSQL on startup, but:

- There's a window between `_engine = ExecutionEngine()` and `_reconcile_from_db(_engine)` where the engine is empty. If any code queries the engine during this window, it sees zero positions.
- `bars_held` is not persisted to DB or loaded during reconciliation. After restart, all positions reset to `bars_held=0`, which delays timeout exits.
- The `_processed_signals` set is loaded from DB, but new signals arriving during restart could be missed.

### 7.3 No Funding Rate Model for Futures — Severity: MEDIUM

Binance Futures charges funding rates every 8 hours (typically ±0.01% but can spike to ±0.5% in volatile markets). For SHORT positions held multiple days, this can be a significant cost (or benefit). The system models zero funding rate.

### 7.4 Position Sizing Ignores Leverage

**File:** `execution/sizer.py`

The sizer computes `size = risk_amount / sl_distance` with a max position cap of 25% equity. For Binance Futures, positions can use leverage. The current sizer doesn't account for margin requirements, liquidation prices, or leverage limits.

### 7.5 Alpaca Paper vs Binance Demo Mismatch

Stocks trade on Alpaca Paper, crypto on Binance Futures Demo. These have completely different:
- Order types available
- Fill semantics
- Fee structures
- Market hours handling
- Position tracking

The training engine treats both identically through `open_training_position()`, which abstracts away these differences. This is fine for paper trading but means the codebase has no framework for handling exchange-specific quirks when going live.

## 8. Monitoring and UI Mismatches

### 8.1 Dashboard Drawdown Calculation

**File:** `monitoring/dashboard_api.py`, line 32

```python
dd_pct = round(pm.total_drawdown * 100, 2)
max_dd = round(max(dd_pct, getattr(pm, '_max_dd_seen', dd_pct)), 2)
```

`_max_dd_seen` is an in-memory attribute. After restart, max drawdown resets to current drawdown. The true historical max drawdown is lost. This is misleading for risk assessment.

**Fix:** Persist max drawdown to DB and load on startup.

### 8.2 Strategy Performance Uses In-Memory Trades

**File:** `monitoring/dashboard_api.py`

The strategy performance endpoint (`/strategies`) computes metrics from `engine.closed_trades`, which loads only the last 200 trades from DB on startup. If a strategy has 300+ historical trades, the performance metrics exclude the oldest 100+.

### 8.3 Training vs Production Engine Confusion

The system has TWO execution paths:
1. `execution/engine.py` → `ExecutionEngine` (the original production path)
2. `training/engine.py` → `TrainingPosition` / `open_training_position()` (the active paper trading path)

The dashboard API queries the `ExecutionEngine` (path 1), but all actual trades flow through the training engine (path 2). This means the dashboard is potentially showing different data than what's actually happening.

## 12. Critical Issues Before Live Trading

These MUST be resolved before any real capital is at risk:

1. **Fee model** — without fees, you don't know if the system is profitable
2. **Kill switch per-asset pricing** — the emergency system must work correctly
3. **ADX fix** — strategy filters and regime detection depend on it
4. **EMA-200 correctness** — bull/crash regime gates depend on it
5. **Restart resilience** — bars_held, max drawdown, signal dedup must all survive restarts cleanly
6. **Exchange-native order tracking** — for live trading, every order must have a confirmed fill from the exchange, not just an assumed fill at bar open + slippage
7. **Real slippage model** — fixed 8 bps is too simple. Slippage should scale with order size relative to available liquidity
8. **Real spread model** — fetch actual bid/ask from exchange or use historical spread distributions
9. **Funding rate tracking** — for any futures position held > 8 hours
10. **Circuit breaker on data staleness** — if candles are > 2 bars old, block all new trades (not just log a warning)
